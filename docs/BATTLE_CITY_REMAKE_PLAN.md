## 1. 产品规划（Product Planning）

### 1.1 核心玩法还原度目标

#### 必须 100% 还原的经典元素（还原度红线）

以下逐条是可验收的硬指标，每条最终都要有对照 FC 原版的验证用例：

1. **地图规格**：13×13 大格（26×26 半格）战场；砖墙以半格为单位被破坏。
2. **基地（老鹰）**：被任何一方子弹摧毁 → 立即全局失败，无论剩多少命。
3. **敌方 4 种坦克**及其速度/耐久/分值差异、重型坦克受击变色（数值表见 §1.4，以考据报告为准）。
4. **每关 20 个敌人、场上同时最多 4 个、顶部 3 个出生点轮转**，出生时闪烁无敌。
5. **闪烁红坦克掉落道具**规律：每关第 **4、11、18** 辆敌人为闪烁坦克；道具是**命中即掉**而非击毁即掉（装甲型命中 1 次掉道具后停止闪烁但仍需打满 4 发），落点取 16 个候选位置之一（附录 A.1）。
6. **玩家 4 级升级路线**（星星驱动：提速 → 双发 → 破钢墙），死亡后等级重置。
7. **5 种地形交互**：砖（可破坏）、钢（仅满级弹可破）、树（遮挡不阻挡）、冰（打滑）、河（坦克不可过、子弹可过）。
8. **铁锹道具**：基地周围砖墙临时变钢墙，倒计时结束**变回时的闪烁过渡**也要还原。
9. **时钟冻结、手雷全屏清敌、头盔无敌**的持续时间与判定细节。
10. **双人同屏**：友军子弹命中队友使其短暂僵直（不掉血）、抢道具、共享敌人配额。
11. **关底结算画面**：逐类坦克计数 × 分值滚动结算、道具 500 分。
12. **35 关经典关卡**全部按原版地图 1:1 内置，循环后敌人组成加压。
13. **手感三要素**：格点吸附转向（半格对齐）、同屏己方弹数上限（1 级 1 发 / 高级 2 发）、子弹与子弹可对消。
14. **原版音效节奏**：待机引擎声/移动引擎声两态切换、开火、命中砖/钢、爆炸、道具音。可用重制音源，但触发时机与节奏还原。

> 手感验收方式：以 FC 模拟器逐帧录像为基准，对移动/转向/开火/子弹速度做帧级对照表（P0 阶段建立，见 §4）。

#### 可以现代化的增强元素（不破坏红线）

| 类别 | 增强项 | 原则 |
|---|---|---|
| 画面 | 高清像素重绘（原版调色板）+ 可一键切换「原版 8-bit 皮肤」 | 默认现代皮肤，原版皮肤是尊重也是卖点 |
| 反馈 | 屏幕轻微震动、履带痕迹、弹壳、得分飘字、命中停顿 1-2 帧 | 全部可在设置中关闭（"纯净原版模式"） |
| 音频 | 重编曲 BGM + 原版芯片音双轨 | 同上可切换 |
| 系统 | 存档/续关、成就、排行、每日挑战、关卡编辑器 | 原版没有的外围系统，不碰核心规则 |
| 操作 | 移动端虚拟摇杆、手柄、按键自定义、辅助选项 | 见 §2 |
| 模式 | 无尽、PVP、编辑器分享 | 见 §1.2 |

**明确不做的"增强"**：自由角度移动（必须 4 向格点）、坦克血条化（保持受击次数模型）、镜头跟随缩放默认开启（默认全图固定视角）。

### 1.2 游戏模式设计

| 模式 | 内容 | 优先级 |
|---|---|---|
| **经典战役** | 原版 35 关 + 循环加压；双人本地合作贯通 | P0 |
| **新战役（复刻+）** | 自制 4 章 × 10 关新地图，引入 BOSS 关（§1.3），使用同一套规则 | P1 |
| **无尽模式** | 波次制，每 5 波三选一局内临时强化；本地家庭榜 | P1 |
| **每日挑战** | 日期种子确定性生成一张图 + 修正条件（如"钢墙×2""禁用铁锹"），全家同题比分，复用现有 `DailyChallenge` 思路 | P1 |
| **双人合作（本地）** | 所有 PVE 模式支持：键盘分区 / 双 Switch 手柄 / 平板双侧触控 | P0（战役内） |
| **对战（本地）** | 2 人同屏对抗小地图（互相摧毁 or 抢基地），亲子对战 | P2 |
| **关卡编辑器** | 网格绘制 + 敌人波次配置 + 本地保存；孩子自己画地图给爸妈打 | P2 |

（v1 的"在线多人 PVP/合作"已裁撤，调研结论留档附录 B.2。）

### 1.3 关卡系统设计

- **关卡数量**：经典 35（手工 1:1 录入，数据文件化）→ 新战役 40 → 编辑器 UGC 无上限。
- **地图生成方式**：
  - 战役关：纯手工数据（JSON/文本网格，一格一字符，git 可 diff）。
  - 无尽/每日：**约束式程序生成**——先放基地与出生点保护区，再按模板块（砖阵/钢十字/河道）拼装，最后用连通性检查（flood fill）保证出生点到基地必有通路；种子来自日期或房间号，保证可复现。
- **难度曲线**（经典战役循环 + 新战役）：
  - 第 1-5 关：教学坡度，基础坦克为主，无冰/河。
  - 第 6-15 关：引入快速/加农坦克、冰面、河流分割。
  - 第 16-25 关：重型坦克占比上升、地形组合刁钻、道具依赖增强。
  - 第 26-35 关：原版后期的高压配置。
  - 循环第 2 周目起：敌人组成整体上移一档（原版行为），第 3 周目封顶。
- **BOSS 关**（仅新战役/无尽，不进经典 35 关）：每章 1 个，如「装甲列车」（沿轨道移动的多节车厢，逐节破坏）、「要塞炮台」（固定多炮塔 + 召唤小坦克）。BOSS 遵守同一物理规则（子弹可对消、砖钢交互），只是体型与行为特殊。

### 1.4 坦克与道具系统

> ⚠️ 下表数值以考据调研报告（附录 A）为最终依据，此处为设计口径；实现时数值全部进配置文件（`content/`），禁止散落在逻辑代码中。

**敌方坦克 4 类 + AI 行为差异**：

| 类型 | 分值 | 特征 | AI 行为倾向 |
|---|---|---|---|
| 基础坦克 | 100 | 慢速慢弹 | 高随机游走，弱目标性 |
| 快速坦克 | 200 | 移速最高 | 强烈偏向基地路线（威胁源） |
| 加农坦克 | 300 | 子弹速度最高 | 中等目标性，远距开火意愿高 |
| 重型坦克 | 400 | 4 次受击（变色反馈） | 稳步推进，压制型 |

AI 架构上四类共用一个状态机（游走/追击/攻基地/避险），用**权重参数**区分类型（转向随机度、目标偏好、开火欲望），周目加压 = 调权重不加代码（§3.6）。

**玩家升级路线**：0 星基础 → 1 星子弹加速 → 2 星双发 → 3 星破钢墙。死亡重置为 0 星（红线）；现代模式可选"死亡保留 1 星"作为辅助选项（默认关）。

**道具系统（考据结论：FC 原版严格为 6 种，"经典 8 种"是跨版本混淆，见附录 A.3）**：

| 道具 | 效果 | 来源版本 | 出现逻辑 |
|---|---|---|---|
| ⭐ 星星 | 升 1 级 | FC 原版 | 命中闪烁坦克掉落，16 候选落点之一 |
| 💣 手雷 | 场上敌人全灭 | FC 原版 | 同上 |
| ⛑ 头盔 | 临时无敌（不保护基地） | FC 原版 | 同上 |
| ⏱ 时钟 | 场上敌人冻结一段时间 | FC 原版 | 同上 |
| 🛡 铁锹 | 基地围墙变钢墙，失效前闪烁变回 | FC 原版 | 同上 |
| 🚗 坦克 | +1 命 | FC 原版 | 同上 |
| 🔫 枪（直升 3 星） | 扩展道具 | ROM 未启用废案（TCRF 考据），Tank Force 等后续作启用 | 仅新战役/无尽启用，经典战役**不出现**（还原度红线） |
| ⛴ 船（可过河面 + 一次性护盾） | 扩展道具 | Game Boy 版（1991） | 同上 |

道具在场上限 1 个，再次命中闪烁坦克会**替换**未拾取的旧道具（原版行为）；拾取 +500 分。头盔/时钟/铁锹的精确秒数公开资料无权威数字（附录 A.9），P0 帧级对照阶段以 FC ROM 实测定值。

### 1.5 进度与存档系统

- **关卡解锁**：经典战役线性推进，通过即解锁下一关；已通关卡可自由选关重打（原版密技的现代化）。战役进度、最高分、每关最佳表现（用命数/剩余时间）分档记录。
- **续关规则**：经典模式失败回本关重开（保留原版"续关不保留分数"的选项开关）。
- **成就系统**：约 40 个，三层——还原向（"一命通 35 关""满级坦克通关"）、技巧向（"一发子弹对消 10 次""铁锹归零前 1 秒守住基地"）、收集向（"8 种道具各拾取一次"）。复用现有 `content/achievements.ts` 模式。
- **排行与档案**：本地家庭榜（复用现有本机荣耀榜）+ **多档案位**（每个孩子独立的进度/成就/皮肤存档）。云榜与云存档裁撤（§0.2）。
- **存档**：复用版本化 `SaveRepository`（LocalStorage 双份安全写），回放与编辑器地图入 IndexedDB（§3.9）。

### 1.6 长期内容与激励（v2：非商业，付费设计整体裁撤）

**原则：所有内容靠游玩解锁，零内购、零广告、零账号。** 解锁体系本身就是给孩子的激励系统：

- 皮肤更衣室：坦克涂装 / 履带痕迹样式 / 爆炸特效 / 基地外观，绑定成就与战役进度解锁（例："通关第 2 章 → 恐龙涂装"）。
- 每日挑战 = 全家同题：同一日期种子，家庭成员轮流挑战比分（本地家庭榜）。
- 新内容节奏跟随开发意愿而非运营日历：新地图/新皮肤随版本更新直接送达（PWA 自动更新）。

### 1.7 目标用户（v2：家庭自用）

| 人群 | 诉求 | 我们的应答 |
|---|---|---|
| 自家孩子（6-12） | 上手即玩、失败不挫败、视觉反馈够炫 | 儿童预设一键档（§2.5）、现代皮肤默认开、成就即奖励 |
| 家长（怀旧玩家） | "和小时候一模一样"，陪玩零门槛 | 还原度红线 + 原版皮肤/芯片音一键切换、本地双人 P0 |

**产品一句话**：给孩子的、帧级还原的坦克大战——爸妈玩到的是童年，孩子玩到的是超越原版的手感与画面。

## 2. 交互与体验设计（Interaction & UX）

### 2.1 控制方案

| 端 | 主方案 | 说明 |
|---|---|---|
| PC 键盘 | P1：方向键/WASD 移动 + J/空格开火；P2：可自定义分区 | 4 向数字输入天然契合原版格点移动；支持按键完全自定义（复用 InputManager） |
| 手柄 | 十字键优先，摇杆量化为 4 向（最近轴向 + 死区）；**Switch 手柄为一等目标**，分级方案见下 | 一对手柄 = 最佳双人体验 |
| 移动端 | **左侧固定式 4 向虚拟十字键（非自由摇杆）+ 右侧大开火键** | 关键决策：Battle City 是 4 向游戏，自由摇杆会产生方向抖动，**量化十字键手感显著更好**；十字键支持斜向容错（45° 分区偏置，倾向保持当前方向） |
| 鼠标 | 不参与战斗（原版无瞄准概念），仅菜单/编辑器 | 编辑器内鼠标绘制网格是主交互 |

**Switch 手柄支持（分两级迭代，可行性已明确）**：

- **第一级（近乎零成本，Phase 2 落地）**：**Switch Pro Controller** 蓝牙配对到电脑/平板后，Chrome/Edge 以标准 Gamepad API 识别（standard mapping），现有 `InputManager` 的手柄归一化直接生效——十字键移动 + A/B 开火即插即玩；iPadOS 16+ 在系统级支持 Pro Controller 与 Joy-Con，Safari 同样经 Gamepad API 暴露。两只 Pro Controller = 双人合作最佳形态。
- **第二级（需专门适配，Phase 3 落地）**：**单只 Joy-Con 横握**（孩子小手最友好，一对 Joy-Con 天然双人）按钮布局非标准且各浏览器映射不一致，需要 Joy-Con 专属映射表 + 进游戏时"按一下开火键"的自动识别校准。体感/HD 震动需 WebHID（仅桌面 Chrome），不列为目标。
- **兜底**：任何未识别手柄提供"按键学习"式手动映射界面，映射结果存入存档。

移动端细节：开火键支持按住连发（受同屏弹数上限约束，不破坏平衡）；虚拟键位置/大小/透明度可调；触控热区下探比可视图形大 20%。

### 2.2 镜头与视角

- **默认：固定俯视全图**（原版红线）。13×13 战场在任何端整图呈现，等比缩放 + 两侧信息栏（敌人余量/命数/关卡号，还原原版右栏）。
- 竖屏手机：战场置上、控制区置下；信息栏压缩为顶部单行。
- 增强反馈（均默认弱、可关）：基地被击中/手雷爆炸时 2-4 帧轻微屏震；玩家被击毁时 100ms 命中停顿。**不做**镜头跟随与动态缩放（破坏全局信息，原版策略性依赖全图视野）。

### 2.3 UI/UX 框架

```
启动页(PWA) → 主菜单 ┬ 经典战役 → 选关 → 战前画面(STAGE N) → 局内 → 结算(逐类计分) → 下一关
                    ├ 无尽/每日 → 修正器说明 → 局内 → 本地家庭榜
                    ├ 双人入口(所有模式内嵌选择)
                    ├ 编辑器 → 画布/测试运行/保存
                    ├ 图鉴与成就
                    └ 设置(音频/画面/控制/辅助/语言)
局内暂停 = 覆盖层：继续/重开/设置速览/退出（暂停时模拟完全冻结，复用 PauseState）
```

- 战前"STAGE N"灰底画面、结算逐行滚动计分——这两个仪式感画面必须还原。
- HUD 遵循原版右栏结构；移动端折叠为角标。
- 皮肤更衣室（全部游玩解锁，无商店概念）入口放主菜单二级，不进局内。

### 2.4 视觉与音效反馈系统

分层反馈预算（每项独立开关，"纯净原版模式"一键全关）：

| 事件 | 视觉 | 音频 | 触觉(移动端) |
|---|---|---|---|
| 子弹命中砖 | 半格碎裂粒子(4-6 片) | 原版"哒" | — |
| 命中钢墙 | 火花 + 弹开 | 金属"叮" | 轻 |
| 坦克被毁 | 两帧圆形爆炸 → 现代皮肤加碎片 | 爆炸低频 | 中 |
| 玩家被毁 | 大爆炸 + 停顿帧 | 专属爆炸音 | 强 |
| 得分 | 原版式分值数字浮现于击毁点（100/200/…） | — | — |
| 履带 | 现代皮肤：淡出履带痕迹（对象池，上限 64 段） | 移动/待机双态引擎声 | — |

### 2.5 无障碍与手残友好

- **「儿童预设」一键档**：慢速 90% + 出生无敌延长 + 死亡保留 1 星 + 大触控热区，设置首屏一个开关搞定（给孩子开、给爸妈关）。

- 减少闪光模式（复用现有设置）：爆炸/道具闪烁降频。
- 手残辅助（默认关，成就在开启时打星号标注但不禁用）：
  - 慢速模式（全局 80% 速度）；
  - 出生额外无敌 +2s；
  - 死亡保留 1 星；
  - 基地被击中一次的"最后机会"缓冲（新战役限定，经典模式不可用）。
- 大字 UI、左右手镜像布局、按住/点按开火切换——均复用星核远征已有的家庭设置面板模式。

### 2.6 多端适配策略

| 端 | 布局 | 目标 |
|---|---|---|
| PC ≥1024px | 战场居中整图 + 左右信息栏，像素完美整数倍缩放（1x/2x/3x） | 60fps 恒定 |

## 3. 技术架构与实现方案（Technical Architecture）

> 本节技术选型的调研验证见附录 B；此处为架构决策。

### 3.1 渲染引擎选型

**推荐：纯 Canvas 2D + 原生 TypeScript（自研轻渲染器），不引入 Phaser/PixiJS。**（最终版本号与生态佐证见附录 B）

理由：

1. **规模匹配**：13×13 tile + 峰值 <10 坦克 + <20 子弹 + 粒子，Canvas 2D 单层 drawImage 即可 60fps，引擎的批处理/场景图收益为零。
2. **还原度需要帧级控制**：像素完美缩放、半格破坏、两帧爆炸动画，自研渲染器控制力最强，没有引擎抽象层的"帮倒忙"。
3. **仓库一致性**：星核远征已验证"确定性 Simulation + 渲染器消费快照"分层；复刻线换一个 `Canvas2DRenderer` 实现同一消费接口即可，架构同构、心智成本最低。
4. **包体积**：目标首包 gzip <150KB（含全部 35 关数据），任何引擎都会翻倍以上。
5. 若后期特效需求膨胀（大规模粒子/光照），迁移到 PixiJS 的成本被"渲染器只消费快照"的边界锁死在一个目录内。

### 3.2 物理与碰撞

不需要物理引擎。全部碰撞为 AABB + 网格：

- 坦克 vs 地形：半格网格查询（坦克占 2×2 半格），移动前查询目标格可通行性；冰面 = 松开方向键后保留 N 帧速度。
- 子弹 vs 地形：射线步进到半格边界，命中砖执行半格破坏写回地图位图。
- 动态实体互撞（坦克/子弹/子弹对消）：实体数 <40，复用现有 `SpatialHash` 甚至朴素 O(n²) 都足够；子弹对消 = 两弹 AABB 相交即同时湮灭。
- 格点吸附：转向时将横/纵坐标吸附到最近半格（原版手感核心，进帧级对照表验收）。

### 3.3 资源加载与管理

- **精灵图**：单张 spritesheet（原版皮肤 + 现代皮肤各一张）+ JSON 图集描述；构建期由脚本从分帧源图拼合（沿用 `scripts/` 资产脚本模式）。
- **Tilemap**：关卡 = 纯文本网格（26×26 字符，`B`砖 `S`钢 `W`水 `T`树 `I`冰），git 可 diff、编辑器可导出同格式。
- **音频**：短音效走 WebAudio 解码缓存；chiptune 音源优先程序化合成（零资源体积，延续现有 AudioManager 路线），BGM 用压缩 ogg/m4a 双格式。
- 加载策略：首屏仅加载当前皮肤 spritesheet + 第一关；其余关卡数据本就内嵌（文本极小），第二皮肤与 BGM 懒加载。

### 3.4 状态管理与游戏循环

**推荐：有限状态机（游戏流程）+ 轻量 ECS-lite（战斗实体），不上完整 ECS 框架。**

- 顶层 FSM：`Boot → Menu → StagePre → Playing ⇄ Paused → StageResult → GameOver`，每态一个对象（enter/update/exit），复用 `Game.ts` 编排模式。
- 战斗层：实体 = 类型化的紧凑结构体数组（tanks/bullets/powerups/effects 各一池），系统函数按固定顺序遍历（input → intent → move&collide → fire → bullets → spawn → powerup → cleanup）。这是"结构体池 + 系统函数"的 ECS-lite，避免完整 ECS 的组件注册开销与心智负担（实体种类就 4 种，完整 ECS 是过度设计）。
- **固定 60Hz 逻辑 Tick + 渲染插值**：`accumulator` 模式；这是回放、每日挑战一致性、未来联机的统一地基（`ARCHITECTURE.md` §3 既定方向）。
- 全局确定性纪律：模拟内禁用 `Math.random`/`Date.now`（现有 RNG + 测试已有先例），浮点统一或直接用整数坐标（半格 ×256 定点数，彻底规避浮点跨端差异——联机 lockstep 的前提）。

### 3.5 地图系统（含动态破坏）

- 双层数据：静态装饰层（树/冰底纹）+ 碰撞层（`Uint8Array(26×26)`，每格一个地形枚举）。
- 砖墙破坏到**1/4 精度**：每砖格 1 字节位掩码记 4 个子块，子弹按打击方向清除半边两子块（原版规则）；渲染层按位掩码选子精灵。
- 钢墙被满级弹整格清除；铁锹效果 = 基地周围 8 格临时改写 + 定时器还原（还原时若有坦克占位，按原版行为处理——考据确认）。
- 脏格重绘：地形画在离屏 Canvas，破坏时只重绘变更格，主循环 blit 一次（§3.8）。

### 3.6 AI 系统架构

**推荐：分层状态机 + 权重参数，不用行为树、默认不用 A\*。**

- 结构：`意图层`（游走/追玩家/攻基地/取道具，按类型权重定期重选）→ `转向层`（原版式随机转向 + 目标方向偏置）→ `开火层`（视线上有可打目标/可破砖时按欲望概率开火）。
- 原版 AI 本质是"带偏置的随机游走"，**刻意不聪明**是玩法的一部分——经典模式禁用寻路，保还原。
- 新战役/无尽的高难敌人可选启用**格点 A\***（26×26 网格上代价可忽略，砖=高代价可破、钢/河=不可走），作为"精英"行为差异点。
- BOSS：独立脚本化状态机（阶段驱动），不塞进通用权重体系。
- 所有权重进 `content/enemies.ts` 式配置，周目加压 = 配置乘数。

### 3.7 网络架构（v2 裁撤）

非商业家庭版不做在线多人与云服务。方案调研保留在附录 B.2 备查；架构上唯一保留的"接缝"是确定性模拟 + 输入帧回放（§3.4）与 `platform/` Provider 协议——将来若想加联机，确定性 lockstep 的全部前提仍是现成的，当期无需为此付任何成本。

### 3.8 性能优化策略

按性价比排序（本项目规模下，多数"高级优化"是负收益，明确不做的也列出）：

| 做 | 说明 |
|---|---|
| 分层 Canvas | 底层地形（离屏缓存 + 脏格更新）/ 中层实体（每帧重绘）/ 顶层 UI-HUD（低频 DOM 或独立 Canvas） |
| 对象池 | 子弹/粒子/飘字/履带痕迹全池化（现有粒子池思路平移） |
| 整数坐标渲染 | 所有 drawImage 目标坐标取整，防亚像素采样模糊与性能抖动 |
| spritesheet 单图 | 消除纹理切换，一次 draw 调用一个源图 |
| 逻辑/渲染分频 | 逻辑锁 60Hz，渲染跟 rAF；后台标签页暂停（复用现有后台暂停流程） |
| PerformanceGovernor 平移 | 帧时间移动平均自动降档（关粒子→关履带→降特效） |

**明确不做**：Web Worker 跑模拟（实体量不值得跨线程序列化成本，除非联机后作为服务器复算复用）、脏矩形动态实体层（全图重绘 26×26 tile 比脏矩形管理更快）、LOD（俯视 2D 无意义）、WebGL 化（除非皮肤系统膨胀出大量粒子需求）。

### 3.9 代码结构与模块划分

```text
src/                 # 复刻线原地接管（逐目录替换星核远征实现，基建目录原样保留）
  core/        # (复用) EventBus/RNG/PauseState + 新增：FSM 流程编排、固定 Tick 循环
  sim/         # 新增：确定性模拟 world.ts / tank.ts / bullet.ts / collide.ts
               # / terrain.ts / powerup.ts / ai/ / rules.ts（零 DOM 依赖）
  content/     # 替换：tanks.ts enemies.ts powerups.ts levels/（35 关文本网格）skins/
  render/      # 替换：Canvas2DRenderer（layers/ sprites.ts effects.ts），移除 Three.js
  audio/       # 复用架构：chiptune 合成 + 现代音银行
  input/       # (复用) InputManager + 新增：TouchDpad、Joy-Con 映射表
  ui/          # 替换：菜单/HUD/结算/编辑器（DOM）
  persistence/ # (复用) SaveRepository（新 schema）+ IndexedDB（回放/自制地图）
  platform/    # (复用) PerformanceGovernor；Provider 协议保留为将来余地
```

核心不变式：**`sim/` 目录零浏览器 API 依赖**，vitest 直测；`render/` 与 `ui/` 只消费 `WorldSnapshot`。CI 增加"确定性回归测试"：固定 seed + 输入脚本跑 N Tick，断言世界哈希（防止无意引入不确定性——回放与每日挑战一致性的长期保险）。替换完成后从 `package.json` 移除 `three` 依赖。

### 3.10 数据持久化

| 数据 | 介质 | 理由 |
|---|---|---|
| 进度/设置/成就/本地榜 | LocalStorage（复用双份安全 SaveRepository，schema 独立于星核远征） | 小、同步读写、迁移体系现成 |
| 回放（seed+输入帧）/ 编辑器地图 | IndexedDB（idb-keyval 起步，UGC 检索需求膨胀再升 Dexie） | 可能较大、可枚举、避免 LocalStorage 无界增长（现有 3600 帧上限教训的正解） |
| 云存档/云榜 | **不做**（非商业家庭版；Provider 协议保留为将来余地，调研留档附录 B.4） | — |

### 3.11 其他技术考虑

- **打包部署**：单入口 Vite 构建 + 现有 GitHub Pages 流程与 PWA 壳直接沿用；家里各设备"添加到主屏幕"即近原生体验，版本更新经 PWA 自动送达。关卡/平衡数值集中在 `content/`，无需热更基建。
- **版本控制与迭代**：沿用现有 conventional commits + PR 流程；`sim/` 的每个规则改动必须附对照测试（还原度回归表）；关卡文本网格 review 时可读 diff 是选型红利。
- **移动端细节**：`touch-action: none` + 多点触控独立 tracking（移动与开火不互斥）；虚拟键渲染在 DOM 层不进 Canvas（利用系统合成器，省重绘）；`visualViewport` 处理刘海与键盘；震动走 `navigator.vibrate` 能力检测。
- **扩展性**：新坦克/新道具/新地形 = `content/` 加配置 + `sim/` 注册一个行为函数 + spritesheet 加帧，遵循现有 ARCHITECTURE.md §4 的内容扩展纪律；关卡即文本，UGC 与官方关同格式。

## 4. 开发路线图（v3：MVP → Alpha → Beta → 正式版）

> 节奏假设：Fable5 负责架构/契约/验收，Sonnet 子代理执行具体任务；每阶段出口都是"孩子可以上手玩"的可运行版本。在线化与商业化已裁撤（愿望清单见文末）。

### MVP · 单人可玩，上线给小朋友体验（~2 周）【当前阶段】

**硬边界：单人、经典第 1-3 关、键盘操作。** 边界之外一律进 backlog。

- 地基：星核远征打 tag `star-voyage-final` 归档（✅ 已完成）；固定 60Hz Tick + 定点数确定性模拟 + 哈希回归测试上 CI；新代码在 `src/classic/` 落地，MVP 完成时切换首页入口。
- 玩法全量规则（这是"体验乐趣"的底线，不裁）：5 种地形交互（1/4 砖破坏、冰面打滑）、玩家 4 级升级、敌方 4 类权重 AI（每关 20 敌/场上 4/闪烁坦克 4·11·18）、FC 6 种道具、基地/出生/结算全流程、计分与奖命。
- 内容：经典第 1-3 关 1:1 录入；原创像素精灵图集（自绘，不用 Namco 素材）；基础 chiptune 音效（开火/命中/爆炸/道具）。
- 外壳：主菜单 → 战前 STAGE N → 局内 HUD → 结算 → GameOver；暂停；本地最高分。
- **出口标准**：孩子能独立从菜单开局、用键盘玩通 1-3 关并看到结算；第 1 关与 FC 录像并排对照通过核心红线（格点吸附/弹数上限/子弹对消/基地判负）；`npm run check` 全绿。
- 数值策略：无权威来源的数值（道具秒数等）用附录 A 口径的临时值，**全部集中在 `constants.ts` 并标注 provisional**，Alpha 阶段实测校正。

### Alpha · 内容全量（~2-3 周）

- 经典 35 关全量录入 + 选关/进度存档 + 成就雏形；触屏虚拟十字键（Pad/手机可玩）。
- 帧级对照表 `docs/fidelity-table.md`（FC 逐帧实测）校正全部 provisional 数值；14 条红线逐条验收。
- 移除星核远征代码与 `three` 依赖，仓库瘦身完成。
- **出口标准**：35 关键盘+触屏贯通；还原度红线 14/14；首包 gzip <150KB。

### Beta · 双人 + Switch 手柄 + 视觉超越（~3-4 周）

- 本地双人（键盘分区 + 友军僵直等规则）；**Switch Pro Controller 第一级支持**（§2.1，含 iPad Safari）。
- 现代高清皮肤 + 双音源银行 + §2.4 反馈全表——"视觉超越原版"的主战场，UI 视觉方案用 **cc-design** 产出后实现；一键切回纯净原版模式。
- 成就、本地家庭榜、多档案位、设置面板（儿童预设 + 无障碍全项）；PWA 部署 GitHub Pages。
- **出口标准**：35 关双人贯通；两只 Pro Controller 同玩验证；移动端 45fps 底线。

### 正式版 1.0（~4-5 周）

- 新战役 40 关 + 2 个 BOSS；无尽模式（程序生成 + 连通性验证）；每日挑战（全家同题同种子）；扩展道具（枪/船）在新模式启用。
- 关卡编辑器（孩子自己画地图）；本地对战小模式；**Joy-Con 单只横握映射 + 手柄按键学习界面**；皮肤更衣室（全部游玩解锁）。
- **出口标准**：全模式回归绿；每日挑战跨设备同种子哈希一致。

### 愿望清单（默认不做，调研留档）

- 在线多人 / 云存档 / 云榜（附录 B.2、B.4 留档；Provider 协议与确定性模拟是预留接缝）。
- Joy-Con 体感与 HD 震动（WebHID，仅桌面 Chrome）。

### 贯穿性优先级原则

1. 还原度红线 > 新功能：任何阶段发现帧级偏差，优先修复。
2. 确定性纪律 > 一切便利：破坏确定性的改动直接拒绝（CI 哈希测试兜底）——它支撑回放、每日挑战与家庭榜公平性。
3. 每阶段末跑全量 `npm run check` + 双端 Playwright 冒烟（复用现有基建）。

| R4 | 手感偏差（格点吸附、连发节奏难以一次调准） | 高/中 | 帧级对照表逐项校验 + 真实用户（孩子）试玩反馈循环；手感参数全部可配置化 |
| R5 | 子代理产出架构漂移、风格不一致 | 中/中 | **契约先行**（`types.ts`/`constants.ts` 由主线亲自定义）；每个子任务附验收清单；集成前跑 `tsc --noEmit` + vitest |
| R6 | Switch 手柄浏览器兼容差异（尤其 Joy-Con 映射） | 中/低 | 分级方案（Pro Controller 标准映射先行）+ "按键学习"手动映射兜底（§2.1） |
| R7 | 范围蔓延（想加的功能太多） | 高/中 | MVP 硬边界（单人/1-3 关/键盘）；红线之外新想法一律进愿望清单，不进当前迭代 |
| R8 | 移动端性能与适配（MVP 后才验证） | 低/中 | MVP 只承诺 PC；PerformanceGovernor 降档机制现成，Alpha 接入触屏时一并压测 |
| R9 | 旧主线替换误删可用基建 | 低/高 | 已打 tag `star-voyage-final`；逐目录替换而非一次删除；每步 `npm run check` 回归 |

## 附录 A：经典机制考据数据（FC/NES 原版，已验收）

> 由 Opus 考据子代理产出、主线验收。来源：Wikipedia、StrategyWiki（含镜像）、Hardcore Gaming 101、TCRF/Data Crystal、GameFAQs、NamuWiki 交叉验证。**无权威公开数字的条目如实标注"未查证"（汇总见 A.9），实现时以 FC ROM 逐帧实测定值，禁止拍脑袋填数。**

### A.1 敌方坦克与闪烁坦克规则

| 类型 | 移动速度 | 子弹速度 | 耐久 | 分值 |
|---|---|---|---|---|
| 基本坦克 Basic | 慢 | 慢 | 1 发 | 100 |
| 快速坦克 Fast | 最快 | 普通 | 1 发 | 200 |
| 火力坦克 Power | 普通 | 快 | 1 发 | 300（归属为推断，见 A.9） |
| 装甲坦克 Armor | 最慢 | 普通 | **4 发** | 400 |

- 装甲坦克受击变色：绿 → 黄 → 浅绿 → 灰，第 4 发击破。
- **约第 30 关后**登场敌人全部为"提速版装甲坦克"（速度升至快速级、仍 4 发耐久、仍 400 分）。
- 闪烁坦克：每关第 **4、11、18** 辆；**命中即掉道具**（非击毁）；装甲型命中 1 次掉道具并停止闪烁，本体仍需打满；道具生成于 **16 个候选位置**之一；场上同时仅 1 个道具，新道具替换旧道具。

### A.2 玩家 4 级升级路线（已三源确认）

| 等级 | 星数 | 能力 |
|---|---|---|
| 1 级 | 0 | 单发、慢弹 |
| 2 级 | 1 | 子弹加速 |
| 3 级 | 2 | 同屏双发 |
| 4 级 | 3 | 可破坏钢墙（且对砖墙双倍破坏范围） |

RAM 佐证（Data Crystal RAM map）：等级状态 `0x00A8` = 0x00/0x20/0x40/0x60。死亡后重置回 1 级，分数与剩余命保留。

### A.3 道具（关键澄清）

- **FC 原版严格 6 种**：星星 / 手雷 / 头盔 / 时钟 / 铁锹 / 坦克(1UP)，拾取一律 **+500 分**。
- **"枪"是 ROM 未使用道具**（道具值 6，TCRF 考据），原版不会出现；**"船"出自 Game Boy 版（1991）**（过水面 + 一次性护盾）。"经典 8 种"系跨版本混淆——本项目将两者定位为**新模式扩展道具**，经典战役不启用（§1.4 已同步修正）。
- 头盔无敌不保护基地；铁锹失效前钢墙闪烁警告后变回砖墙。

### A.4 关卡结构

- 35 关，通关后循环；**第 70 关后重置回第 1 关**，仅保留强化与分数。
- 每关 20 辆敌人；场上同时上限 4 辆（攻略共识）；顶部左/中/右 3 个出生点。
- 地图 **13×13 大格**（Wikipedia 逐字）；26×26 半格制是由砖墙半格破坏机制反推的工程惯例（复刻界通行做法），坦克占 2×2 半格。
- 敌我出生均有闪烁无敌，时长未查证（A.9）。

### A.5 地形交互（5 种）

| 地形 | 坦克 | 子弹 | 规则 |
|---|---|---|---|
| 砖墙 | 阻挡 | 吸收并破坏半格 | 3 星弹双倍破坏；**残墙陷阱**：同面打 3 次形成"单侧打不穿但仍挡坦克"的残墙（可造陷阱，需还原） |
| 钢墙 | 阻挡 | 普通弹被挡 | 仅 3 星玩家弹可毁；敌人永远打不掉 |
| 树林 | 通过 | 穿过 | 遮挡坦克与子弹视觉（埋伏点） |
| 冰面 | 通过（打滑惯性） | 穿过 | 松开方向后滑行一段 |
| 河流 | 阻挡 | 穿过 | 原版无船 → 绝对屏障 |

### A.6 计分与奖励

- 击破 100/200/300/400；道具 +500。
- **FC 版每 20,000 分 +1 命**（街机 VS. 版为 20,000/70,000，注意区分）。
- 关底结算：逐类型"击破数 × 分值"滚动统计。
- 双人结算：该关击破更多的一方**额外 +1000 分**。

### A.7 双人模式差异

- FC 原版独有（街机原版为单人交替）；P1 黄 / P2 绿。
- 敌人配额共享（仍 20 辆）；道具先到先得。
- **友军伤害**：命中队友使其**定身一段时间（仍可开炮，不掉命）**。

### A.8 易忽略细节清单（还原度验收项）

1. 基地被毁立即 Game Over，**玩家自己的子弹也能打爆自家基地**。
2. 玩家死亡星级清零；分数/命保留。
3. 铁锹到期前闪烁警告。
4. 时钟冻结场上已存在敌人（是否波及冻结期间新出生者未查证）。
5. 冰面惯性滑行、树下埋伏、残墙陷阱（A.5）。
6. 道具替换机制可能"刷掉"没吃到的好道具。
7. 敌人 AI = 伪随机转向 + 整体向下（基地方向）压进（观察性共识描述，非官方文档）。
8. 30 关后全高速装甲坦克（A.1）。

### A.9 未查证条目（P0 阶段以 ROM 逐帧实测补齐，进 fidelity-table）

1. 头盔/时钟/铁锹的精确持续秒数（公开资料仅 "a period of time"）。
2. 各坦克与子弹的精确速度（像素/帧）。
3. 出生无敌闪烁精确时长。
4. "场上上限 4、顶部 3 出生点"为攻略共识，未见权威页逐字定量。
5. 300 分归属火力坦克为推断（四档分值本身确定）。
6. 时钟是否冻结新出生敌人；手雷清敌是否计分。
7. 26×26 半格制为工程反推（官方仅 13×13）。

> Data Crystal 的 Battle City RAM map 对计时类地址标注不完整，上述数值需 P0 阶段用模拟器逐帧录像 + RAM 观测定值，结论写入 `docs/fidelity-table.md` 作为唯一事实源。

## 附录 B：技术选型调研佐证（2025-2026 现状核查，已验收）

> 由 Opus 调研子代理产出、主线验收。事实全部采信；两处**推荐结论被验收方裁定调整**，差异与理由标注在对应小节。

### B.1 渲染引擎现状（2026-07 核查）

| 引擎 | 最新状态 | 包体积 (gzip) | 备注 |
|---|---|---|---|
| 纯 Canvas 2D + TS | 浏览器原生 | 0 KB | 无生态，需自研渲染器 |
| PixiJS | v8（最新 8.19.0），月度发布，WebGPU/WebGL 双后端 | 核心约 150 KB，可 tree-shake | 全库 TS，社区活跃 |
| Phaser | **Phaser 4 已于 2026-04 正式发布**；3.90 为 v3 末版 | 默认约 310–500 KB | 生态与文档第一，渲染与 update 循环耦合较深 |
| Kaplay / Excalibur | 活跃但偏原型/中小项目 | 小-中 | 渲染基准与社区规模不足以支撑长期迭代 |

调研核心结论（采信）：**本项目规模（13×13、<100 实体）下各方案渲染性能无实际差异**，区分维度是 TS 原生性、包体积、逻辑与渲染的解耦程度。

**裁定差异**：调研推荐 PixiJS v8 为首选（理由：为联机预留渲染/逻辑解耦架构）。验收方维持 §3.1 的**纯 Canvas 2D 首选**：解耦是我们架构（Simulation 快照消费）自带属性，与是否用 Pixi 无关；在无性能差异的前提下，0 依赖、帧级控制力与仓库"程序化资产、零重型依赖"的既有纪律权重更高。**PixiJS v8 为正式升级备选**：一旦皮肤/粒子特效需求超出 Canvas 舒适区（大量动态光效、数百粒子常驻），沿"渲染器只消费快照"的边界整体替换。

来源：pixijs.com/blog/pixi-v8-launches · phaser.io/news/2026/05/phaser-3-vs-phaser-4 · github.com/Shirajuki/js-game-rendering-benchmark

### B.2 联机方案现状（2026-07 核查；v2 起在线化已裁撤，本节仅存档备查）

| 方案 | 状态 | 要点 |
|---|---|---|
| Colyseus | **0.16（2025-02 发布），MIT 免费**，可自托管；Colyseus Cloud $15/月起 | 房间/匹配/Schema 增量状态同步开箱即用 |
| socket.io / 原生 WS | 活跃 | 裸传输层，同步模型全自研 |
| WebRTC DataChannel | 活跃 | P2P 低延迟（约省 10–15ms），但 NAT/信令复杂、无权威端防作弊差 |
| PartyKit | 基于 Cloudflare Durable Objects | 绑定 CF 生态 |
| Nakama | 活跃，权威服务器 + 全家桶 | 对本项目过重 |

**裁定差异**：调研推荐"Colyseus + 服务器权威 + 客户端预测/和解"，并以"输入延迟、慢客户端拖累全房、适合 RTS/格斗"否定 lockstep。验收方裁定为**分场景**（§3.7）：

- 调研的反 lockstep 论据未计入本仓库关键资产——确定性模拟、定点数、输入帧回放已是既定架构（排行榜防作弊也依赖它），lockstep 的最大成本（实现确定性）在这里趋近于零，而服务器权威 + 预测/和解需要额外实现快照插值与误差修正，成本反而更高。
- **合作 PVE（4 人房间）**：lockstep + WS 中继，2-3 Tick 输入缓冲，慢客户端拖累在合作场景可接受（与好友开黑，非匿名竞技）。
- **匿名竞技 PVP**：采信调研——升级为服务器权威（复用 sim 包在 Node 复算）+ 客户端预测；此时 Colyseus 0.16 作为房间/匹配/传输层是合理选型（MIT、自托管零许可成本）。

来源：colyseus.io/blog/colyseus-016-is-here · docs.colyseus.io/state · developers.rune.ai/blog/webrtc-vs-websockets-for-multiplayer-games

### B.3 构建与部署（采信）

- **Vite 7.0（2025-06 发布）**：要求 Node 20.19+/22.12+；Rolldown 打包器推进中。本仓库已在 Vite 7 系（`vite@^7.1.3`），无迁移成本。
- 托管硬约束：**GitHub Pages 纯静态无 WS**；**Vercel Serverless/Edge 不支持持久 WebSocket**；Cloudflare Pages + Workers/Durable Objects 是三者中唯一原生支持有状态实时后端的。
- 结论（与 §3.11 一致）：静态客户端留 GitHub Pages（现状零成本）；P3 联机服务端独立部署（VPS/Fly.io/Colyseus Cloud），**绝不放 Vercel/Pages**。

来源：vite.dev/blog/announcing-vite7 · github.com/vercel/community/discussions/422

### B.4 数据持久化（采信）

- IndexedDB 封装：**idb-keyval 6.3.0**（get/set 仅约 295 字节 brotli）——回放/编辑器地图存储用它起步；查询/索引需求膨胀（大量 UGC 地图检索）再升级 **Dexie.js**，不要一开始就上（过度设计）。
- 云存档：**Supabase 免费层**（500MB Postgres、API 无每日上限）优于 Firebase Spark（每日读 50K/写 20K 硬上限，对游戏高频小写入不友好）。**注意：Supabase 免费项目 7 天无活动自动暂停**，正式上线需 Pro（$25/月）或保活。
- 小数据（进度/设置/成就）维持 LocalStorage + 现有双份安全 SaveRepository（§3.10 不变）。

来源：npmjs.com/package/idb-keyval · freetiers.com/blog/firebase-vs-supabase-comparison
