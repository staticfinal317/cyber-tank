# ADR 0001：采用 Three.js 分层渲染而非继续扩展单文件 Canvas

状态：已接受

## 背景

原型使用单 HTML Canvas，可以快速验证射击与波次，但难以稳定支持 3D 立体场景、真正的后处理、多主题资产、手柄/触控并行输入、长期成长和可测试的回放。

## 决策

采用 Vite + TypeScript + Three.js。保留纯数据 `Simulation`，Three.js 仅负责渲染；UI 继续使用轻量 DOM，避免把高频 HUD 文字绘制进 WebGL。旧版单文件保存在 `legacy/`，不参与生产构建。

## 结果

优势：

- 可以使用标准 PBR 材质、灯光、阴影和 EffectComposer 后处理。
- 玩法可单测，未来可移入 Worker 或服务器。
- 场景与装备数据可以独立扩展。
- DOM HUD 保持无障碍和响应式能力。

代价：

- Three.js 增加约 156KB gzip 首包。
- 移动端必须维护画质档位和 GPU 资源生命周期。
- 需要明确同步层，避免渲染对象成为玩法权威状态。

## 被否决方案

- 继续单文件 Canvas 2D：无法经济地达到需要的 3D 光照与后处理质量。
- 首版直接采用完整 ECS 与物理引擎：对当前实体量过度设计，会降低儿童手感迭代速度。
- 首版直接上 WebGPU：设备和浏览器覆盖风险高，WebGL2 更适合当前发布范围。
